Sales Performance Dashboard

SQL · Power BI · DAX

The question

A retail manager opens one screen on Monday morning and needs to know: where is revenue won, where is it lost, and what needs attention this week? This dashboard answers exactly that — no chart variety for its own sake.

The approach

  • Shape raw retail transaction data with SQL — aggregations, time-windowed comparisons, and top-N rankings, with every query documented in a commented analysis.sql.
  • Model the result in Power BI with DAX measures (not calculated columns) for revenue, profit, and order KPIs.
  • One disciplined layout: a KPI row, a revenue trend line, a category comparison, and a drill-down detail table.

What it will show

Which product categories drive and drag revenue, regional and segment profitability, seasonality patterns, and the outliers worth investigating — each visual mapped to one of the business questions defined before any chart was built.

Exploratory Data Analysis in Python

Python · Pandas · Jupyter

The question

Given a real-world dataset, what actually drives the outcome — and just as importantly, what can the data not tell us? This project is an investigation, not a gallery: every chart earns its place by moving the answer forward.

The approach

  • Profile the raw data first — shape, types, missingness — before touching anything.
  • Clean with documented decisions: every drop, fill, or type fix justified in a markdown cell, because the reasoning is the analyst work.
  • Investigate with groupbys, distributions, and comparisons, building toward a conclusion rather than a pile of plots.
  • Close with an honest conclusion: findings, confidence, and what I'd examine next with more data.

What it will show

A notebook readable as a story even if you skip the code — demonstrating investigation, scepticism, and clear written reasoning beyond what BI tools show.

Data Migration Validation Toolkit

SQL · Python · Data Quality

The question

After a system migration, how do you prove the data arrived correctly? This toolkit simulates a migration gone subtly wrong — and catches every fault. It's built directly from my experience supporting the data workstream of a live CRM migration covering 500K+ records.

The approach

  • Generate a realistic synthetic source dataset (thousands of customer records via Python + Faker) — no real company data anywhere near this project.
  • Produce a deliberately flawed "migrated" target: dropped rows, duplicates, truncated values, corrupted dates, silent value changes.
  • Build the reconciliation suite that detects each fault class without peeking at the answer key: row-count reconciliation, duplicate detection, referential completeness, format and integrity checks, and value-level source-to-target comparison.
  • Output a validation summary report — the same artifact a real data team produces at cutover.

What it will show

Professional data-quality methodology most junior portfolios never touch: completeness, uniqueness, integrity, and accuracy checks, written up as a reusable validation playbook.
